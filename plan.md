# アプリケーション概要

**キャッチコピー:** 「進捗を、会場の景色に変える」

メンバーがスマホから送る「できた！」という報告が、リアルタイムにメインスクリーンのサクラを成長させ、会場全体を一つのアートにする進捗可視化ツール。

---

## 想定規模

| 項目 | 値 |
|------|-----|
| 同時接続 | **最大 100 人** |
| 投稿数 | **1 日あたり約 300 件** |

この規模は Firestore の読み書き・リアルタイム監視の一般的な利用範囲に収まりやすい。チーム別集計は後述のとおり NoSQL でも実現可能。

---

## MVP と初回リリースの範囲

- **初回リリースに含めないもの:** AI お題、ポーズ撮影など、当初メモにあった **拡張ポイントはすべて対象外**。
- **MVP のビジュアル:** 最初は **花のついていない木（幹と枝のみ）**。投稿があるたびに **その木に桜が追加されていく**（桜が蓄積されること自体がコア）。複雑なカード演出は必須としない。
- **バリデーション:** **当面は設けない**（文字数・必須チェックなどは後からでも可）。
- **認証:** **誰でも投稿可能**だが、**匿名**とする（個人を特定するログインはしない）。

## ルーティング案

| パス | 画面名 | 主な機能・役割 |
|------|--------|----------------|
| `/` | トップ | 班の選択、全体の進捗の確認、`/post` への導線 |
| `/post` | 進捗作成 | 進捗内容の入力、チーム選択、送信 |
| `/screen` | メディアアート | プロジェクター用。`p5.js` による全画面演出。Firestore を監視し、投稿に応じて桜を追加。画面は **一般的な横長** を想定 |

---

## 設定ファイル（いつでも変えられるように）

変更頻度が高い・イベントごとに変えたい値は、**リポジトリ内の設定ファイル**にまとめ、開催前に編集してデプロイする運用を想定する。

- **実体:** `config/settings.ts`（チーム一覧・想定規模・画面アスペクトの目安など）
- **チーム:** **アルファベット表記**で、**A から始め、必要な数だけ**チームを用意する（例: `A`, `B`, `C` …）。配列を編集すればチーム数を増減できる。
- 将来、環境変数や Firebase Remote Config に移してもよい。

---

## 技術スタック（ライト版）

- **Frontend:** Next.js (React) + Tailwind CSS  
  - `/post` はスマホでの「使い心地」重視。  
  - `/screen` は Canvas 要素をフルスクリーンで配置。**横長レイアウト**。
- **Visual:** p5.js  
  - React のコンポーネントとして埋め込み、Firestore のイベントで描画を更新する。
- **Backend / DB:** Firebase (Cloud Firestore)  
  - NoSQL でスキーマレスに管理。チーム別集計も可能（下記「NoSQL と集計」）。
- **Infrastructure:** Firebase Hosting  
  - `output: 'export'` による静的デプロイ。

## リポジトリ・ファイル構成（案）

**Next.js の App Router** を前提に、ルートと責務がぶつからないように分ける。**`create-next-app` 後に `src/` あり／なしで生成される場合があるので、`app/` は `src/app` かルート `app/` のどちらかに統一**すればよい。

```
sakurasake/
├── plan.md
├── config/
│   └── settings.ts              # イベント設定（チーム一覧・規模・画面目安）※既存
├── public/                      # ファビコン、固定画像など
├── firebase/                    # Firebase CLI でデプロイするルール・インデックス（任意）
│   ├── firestore.rules
│   └── firestore.indexes.json
├── src/
│   ├── app/
│   │   ├── layout.tsx           # 共通レイアウト・フォント
│   │   ├── page.tsx             # / トップ
│   │   ├── post/
│   │   │   └── page.tsx         # /post
│   │   └── screen/
│   │       └── page.tsx         # /screen（p5・全画面）
│   ├── components/              # 画面横断 UI（必要なら ui/ 以下に分割）
│   │   └── …
│   └── lib/
│       ├── firebase/
│       │   ├── client.ts        # ブラウザ用 Firebase 初期化（initializeApp 等）
│       │   └── posts.ts         # posts の購読・追加・where('team','==',…) のラッパ
│       └── …
├── .env.local                   # NEXT_PUBLIC_FIREBASE_* 等（リポジトリに含めない）
├── next.config.ts
├── package.json
└── tsconfig.json
```

| 置き場所 | 役割 |
|----------|------|
| `config/settings.ts` | **イベントごとに編集**する値（チーム数・ラベル）。アプリから import。 |
| `src/lib/firebase/` | Firestore へのアクセスを **1か所に集約**（ページ直書きを避ける）。 |
| `src/components/` | フォーム、スクリーン用キャンバスラッパなど **再利用 UI**。 |
| `firebase/*.rules` | **Security Rules**（匿名投稿の許可範囲をここで管理）。 |
| `firebase/*.indexes.json` | `where` + `orderBy` 用の **複合インデックス** をコード化してデプロイ。 |

**まだ作っていないファイル**（`next.config.ts` 等）は、`create-next-app` 実行後に上記のように配置していく想定でよい。

## データベース設計（Firestore）案

### `posts` コレクション（投稿データ）

| フィールド | 型 | 説明 |
|------------|-----|------|
| `id` | string | ドキュメント ID |
| `team` | string | 英字。**設定の `TEAM_LABELS` と整合する値**（例: `A`, `B`） |
| `userName` | string | 投稿者表示名（匿名運用でもニックネーム等は入力可とする想定） |
| `message` | string | 本文 |
| `type` | string | 将来の演出分岐用。MVP では固定でも可 |
| `timestamp` | serverTimestamp | 投稿時刻 |

`id` は Firestore が自動採番してもよい。

## NoSQL（Firestore）でチーム別集計はできるか

**できる。** 方針は次のような選択肢がある。

1. **クエリ + 集計（書き込み時にドキュメントを増やさない）**  
   - `posts` に `team` フィールドを必ず入れる。  
   - **チームごとの件数:** `where('team', '==', 'A')` に対する **count 集計**（Aggregation query）や、クライアントでスナップショットを購読してチーム別に数える（件数が少ないうちは現実的）。  
   - 1 日 300 件・同時 100 人程度なら、設計次第で **追加のサーバーレス無し**でも運用しやすい。

2. **集計用ドキュメントを別コレクションに持つ（読み取り最適化）**  
   - 例: `teamStats/{teamId}` に `count` を保持し、**Cloud Functions** で `posts` の作成時にインクリメント。  
   - ダッシュボードや `/` の「チーム別スコア」表示を **常に O(1) で読む**とき向き。

3. **クライアントで全件スキャンしてグルーピング**  
   - イベント 1 日分の件数が限定的なら、**メモリ上で `team` ごとに reduce** するだけでも可。規模が増えたら 1 か 2 に移行。

**結論:** NoSQL だから集計できないわけではない。**`team` をフィールドとして持ち、クエリまたは集計用ドキュメントで数える**のが定石。MVP は 1 または 3 で足りることが多い。

## 演出・ビジュアル要件

### 1. 基本コンセプト

「個人の進捗が、会場の景色を創る」— 進捗を桜として木に蓄積し、会場の一体感を視覚化する。

### 2. MVP の画面イメージ（`/screen`）

| 要素 | MVP の内容 |
|------|------------|
| **木** | 幹と枝のみから開始。花はゼロ。 |
| **桜** | 投稿があるたびに、木の上に桜が **追加されていく**（増えていくだけがコア）。 |
| **レイアウト** | **横長**（プロジェクター想定。具体比率は `config/settings.ts` の `SCREEN` を参照）。 |

### 3. 桜の描画とデータの対応（仕様を固める）

**結論:** 桜の個数を **別ドキュメントの `count` だけ** に頼らず、**`posts` コレクションのドキュメントを正（ソース・オブ・トゥルース）** とする。

| 項目 | 仕様 |
|------|------|
| **1 桜と 1 投稿** | 原則 **投稿ドキュメント 1 件 = 桜 1 つ**（MVP）。 |
| **桜の「個数」** | **対象クエリに含まれる `posts` の件数** と一致させる。別途「集計用の数値だけ」を正としない（手動 increment はズレやすい）。 |
| **初回表示・リロード** | `posts` を **取得または購読**し、得られた **ドキュメント数ぶん** 桜を描画する。リロード後も **DB の件数と画面上の桜の数が一致**する。 |
| **リアルタイム更新** | `onSnapshot` 等で **追加されたドキュメント** を検知したら、その分 **桜を 1 つ追加**する（または都度「全件から再計算」でも可・MVP 規模ならどちらでもよい）。 |
| **Firestore の `count()` 集計** | **表示用の件数確認**（例: トップの数字）には使える。桜の **座標リストのソース** は引き続き **ドキュメント一覧** を推奨（どの投稿がどの花か対応づけやすい）。 |
| **集計用ドキュメント（例: `stats/total`）** | **必須ではない。** 読み取り最適化で導入する場合は **Cloud Functions + Transaction** などで `posts` と矛盾しないようにする。MVP は **posts のみ** でよい。 |

**クライアントの「記憶」:** ブラウザが保持するのは **描画用のキャッシュ**（座標・見た目の状態）でよい。**永続の正** は常に Firestore の `posts`。ページ再読み込みで **必ず DB から復元**できること。

**未決（あとで決める）:** 桜の **画面上の位置**（枝へのマッピング・乱数シードを `postId` から生成するか等）は実装時に固定する。

### 4. 将来拡張（初回リリース・MVP では必須としない）

以前検討していた **スポットライト（10 秒表示）→ Shrink & Fly → 枝先への定着** などの多段アニメは、桜の蓄積が安定してからでもよい。必要になったら本節に「採用する／しない」を明記して追記する。

## まだ決めておくとよいこと（現時点で足りない情報）

- **Firestore Security Rules:** 匿名投稿を許可する場合の **スパム対策**（レート制限はルール単体では難しく、**App Check** や **Cloud Functions**、または「イベント用の簡易トークン」を検討するか）。
- **`/post` の `userName`:** 完全匿名（固定表示「ゲスト」）か、**任意のニックネーム必須**か。
- **複数投稿が短時間に重なったとき:** `/screen` で **キューするか**、**最新だけ反映するか**（MVP は「すべて桜として反映」でよいが、通知 UI を出すなら要検討）。
- **チーム別「得点」の定義:** 投稿 1 件 = 1 点でよいか、**`type` による重み**を将来入れるか（今は集計の仕様だけ決めると実装が楽）。
- **静的エクスポート（`output: 'export'`）と Firebase の組み合わせ:** ビルド時とランタイムの制約（例: サーバー専用 API の有無）をプロジェクト作成時に一度確認する。
