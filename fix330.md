## 1) 桜の増減ロジックがわからない（今ってDBと連動してる？）

- **結論**: 現状は「桜（blossom）」はフロント（SakuraPulse）側の `submissions` 件数ベースで計算されており、Sinatra + Postgres の `feed_items` とは **別系統**に見える。
- **補足**:
  - SakuraPulse の桜スコアは「投稿数（submissions数）に比例して増える」だけで、**減衰ロジックは現状不要（＝現時点の要件でもなし）**。
  - SakuraPulse の投稿データソースは Supabase（`submissions` テーブル）またはローカル同期（localStorage）。
  - 一方サーバ側（Sinatra）は Postgres の `feed_items` に投稿/コメント/いいねを保存し、`/feed` で返す & WebSocket でスクリーンへ通知する。

### 根拠（参照）

- **桜スコア算出**: `SakuraPulse/src/lib/blossom.ts` の `calculateBlossomScore`
- **データソース**: `SakuraPulse/src/lib/supabase.ts`, `SakuraPulse/src/lib/useRoomSubmissions.ts`
- **サーバDB連動**: `server/app.rb` の `/feed`, `/post`, `/comment`, `/like`

### TODO（本プロジェクト側）

- **減衰ロジックは不要**（仕様として採用しない）
- **桜が何個（=DBの何件）で満開か**を、本プロジェクトのDB（`feed_items`）基準で定義する
  - 例: `kind='post'` の件数で数える / `post+comment` 合算で数える / `likes` も加点する、など
- **DBと連動**する（画面に出る桜/満開判定は `feed_items` を正にする）

### 参考: SakuraPulse 側の「満開＝何個」根拠（コード）

- SakuraPulse の桜スコア: `blossomScore = submissions.length * (COMMENT_SCORE + MOOD_SCORE)`
  - `COMMENT_SCORE = 3`, `MOOD_SCORE = 1` なので `blossomScore = submissions.length * 4`
  - 参照: `SakuraPulse/src/lib/blossom.ts`（`calculateBlossomScore`）
- 满開判定（%）: `blossomPercent = clamp(score / MAX_VISIBLE_BLOSSOM_SCORE, 0, 1)`
  - `MAX_VISIBLE_BLOSSOM_SCORE = 100`
  - 参照: `SakuraPulse/src/constants/blossomStages.ts`
- よって `blossomPercent >= 1` になるのは `score >= 100`:
  - `submissions.length * 4 >= 100` → `**submissions.length >= 25`**
- また `PetalShower` の有効化も `stats.blossomPercent >= 1` なので、同じく **25 submissionsで満開演出**。
  - 参照: `SakuraPulse/src/components/screen/SakuraScene.tsx`（`<PetalShower active={stats.blossomPercent >= 1} />`）

※ここでの「桜が何個」は、実装上は `**submissions` 1件=1カード/1投稿として桜スコアが増える単位**。
※本プロジェクト側は `feed_items` 基準に合わせて、満開閾値を再定義する。

## 2) 大量投稿が来たらまずそう（DB性能 + 見た目）

### 2.1 DB性能（サーバ/DB）

- `server/app.rb` の `/feed` は `ORDER BY created_at DESC LIMIT $1`。大量データ時はインデックス次第で重くなる可能性。
- seed は現状少量のみ。**大量投入用 seed**（例: 1万件）で検証したい。

#### 現状の seed

- `server/db/seeds.rb`（`feed_items` を TRUNCATE → 数件INSERT）

#### TODO（性能）

- **大量seed投入（件数可変）**を用意して、`/feed?limit=...` の応答時間・DB CPU/IO を計測
- `feed_items(created_at)` 等の **インデックス要否**を検討
- WebSocket 通知の多発を想定し、クライアント側の描画/更新負荷も計測（間引き/バッチ化/上限）

### 2.1.1 DB重複（同一内容の投稿は許可される？）

- 結論: **許可される（重複を排除する仕組みは、少なくとも現状のINSERT/テーブル定義からは見当たらない）**。
- 根拠:
  - `feed_items` テーブルにはユニーク制約（例: `(kind, who, content)` のUNIQUE）がない。
    - 参照: `server/db/migrate/001_feed_items.sql`
  - `/post` は `INSERT INTO feed_items (kind, who, content, img, likes)` をそのまま実行し、既存の同一内容を探して差し替える処理がない。
    - 参照: `server/app.rb` の `/post`
  - `/comment` も同様に `INSERT INTO feed_items ...`。
    - 参照: `server/app.rb` の `/comment`
- 補足（いいねの挙動）:
  - `/like` は `kind='post'` かつ `who/content` が一致する **最新のpostだけ** を `UPDATE likes = likes + 1` している。
  - つまり「投稿の重複自体」は許可されるが、いいねは「同一内容の最新投稿」に寄る仕様。

### 2.2 見た目（3Dカード配置）

- SakuraPulse は表示を **最大18件にスライス**しているため、無限にカードが増えて崩壊するタイプの問題はある程度回避できている。
  - `SakuraPulse/src/components/screen/FloatingComments.tsx`（`slice(0, 18)`）
- ただし大量投稿時に「増えたこと」をどう表現するか（密度、粒子化、集約表示など）は別途設計が必要。
- `index.html` 側（Sinatraのスクリーン）は別実装で orbit 要素/トースト等があるため、そこで大量表示した場合の重なりや負荷も要確認。

#### TODO（見た目）

- **大量投稿時は全部をカードにしない**方針を決める（集約表示、粒子/光点、フェードアウト、LOD）
- 表示数上限、時間窓（直近N分）、優先度（最新/いいね多い/投稿のみ等）を設計

### 2.2.1 「スライス」とは何？

- 現状、桜の3Dカード表示は **二段階で上限がかかっている**。
  - `useRoomSubmissions` の内部で、Realtime/ローカルを含めて `mergeSubmissionLists(...).slice(0, 80)` している。
    - 参照: `SakuraPulse/src/lib/useRoomSubmissions.ts`
  - そのうえで実際の `CommentBlock` 表示は `FloatingComments` 側で `slice(0, 18)`。
    - 参照: `SakuraPulse/src/components/screen/FloatingComments.tsx`
- なので「200ぐらい表示してみたい」は、まずは `**FloatingComments` の `slice(0, 18)` を増やす**のが実験ポイントになりそう。
- ただし表示数を増やすほど `CommentBlock` が各アイテムごとに `useFrame`/`Html` を持つため、FPSや端末負荷が増える可能性があるので、`200`は性能測定前提で。

## 4) moreポイント: 任意の画像も投稿できたらいい

### 現状

- Sinatra 側は `img` を **URL文字列として受け取る**だけ（`http/https` のみ + 長さ制限）。
  - `server/app.rb` の `validate_img_url`
- ファイルアップロード（multipart等）は未実装。

### 案（/upload）

- `/upload` を用意し、画像をアップロード → サーバが保存 → 公開URLを返す（返したURLを `img` として既存の投稿フローへ連携）
- 保存先候補:
  - S3 / Cloudflare R2 等 + 署名URL
  - Supabase Storage（SakuraPulseがSupabase依存なので統一のメリットあり）

### 方針メモ（候補）: Cloudinary が濃厚

- 画像は Cloudinary にアップロードして **公開URL** を得る
- 得たURLを `feed_items.img` に保存し、既存の表示（`img` URL前提）に載せる
- 実装案:
  - `GET /upload` でアップロードフォーム表示
  - `POST /upload` でファイル受け取り → Cloudinaryへアップロード → URL取得
  - URLを使って `feed_items` に `kind='post'` としてINSERT（または専用kindを追加するか検討）
  - WebSocketでスクリーンへ通知（現行の `commit_state!` に乗せる）

### 追加で検討したい点

- URL直貼り投稿も残すなら、SSRF/不正URL/巨大ファイル対策（許可ドメイン、サイズ制限、プロキシ等）
- Glide からの送信方式:
  - multipart
  - base64
  - 先にストレージへ直PUTして **URLだけ**を従来通り送る（クライアント直送）

### TODO

- 画像の保存場所（Supabase Storage に寄せるか、Sinatra側でS3等に寄せるか）を決める
- Glide からの送信方式（multipart / 直PUT / base64 等）を決める

### 4.1 画像リンクだけ許可は不便？（Whisk画像など）

- `server/app.rb` の `validate_img_url` は `http/https` のみ許可しているので、ローカル生成画像の扱い次第ではリンクだけだと面倒になりうる（例: `data:` や一時URLなど）。
- そのため「画像投稿フォーム」実装では以下を両立させる方針が候補:
  - URL貼り付け投稿（現状維持）
  - ファイルアップロード投稿（`/upload` で画像を取り込み、URL化して `img` に反映）
